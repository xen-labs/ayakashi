TECH STACK

Frontend (ayakashi):
- Next.js (App Router), TypeScript, Tailwind
- shadcn/ui is fine for primitives if you want them, keep it consistent

Backend (astral-api):
- Express or Fastify (your call, pick whichever you're faster in),
  TypeScript, Mongoose
- deploys separately from the frontend — Railway or Render free tier to start 

Auth: JWT, httpOnly cookies, not localStorage. Our players are on phones clicking bot-generated links in random WhatsApp groups — XSS surface is higher than a normal SaaS signup, cookies are the safer default here.
Short-lived access token + longer refresh token pair, since players will go days/weeks between visits. astral-api sets/clears the cookies; the frontend never manually touches raw tokens in JS.


REGISTRATION PAGE

Flow: player DMs the bot ".register" or ".link" on WhatsApp groups, bot replies with a link to this page carrying a short-lived token in the URL (query param, something like ?rt=xxxx) identifying which WhatsApp number is registering. Frontend just needs to carry that token through to the final submit — API will validate it.

Fields:
- phone number — country flag picker with search, this is the biggest ux detail to get right, most players won't know their country code. Use react-phone-number-input or similar, don't build this from scratch
- password (with confirm; strength meter nice-to-have, not required)
- age 
- "remember me" checkbox
- forgot password link → /forgot-password

Forgot password flow:
1. /forgot-password page, single button "Continue via WhatsApp"
2. opens wa.me/<botnumber>?text=recover (prefilled message) — this leaves our app entirely
3. bot sends either an OTP to enter on a page we host, or a magic link that auto-verifies — undecided which, build the OTP-entry page (/reset-password?token=xxx) first since it's the safer bet  architecturally, magic link can reuse most of the same UI later

ALT-ACCOUNT 

We need to prevent people from registering multiple accounts to abuse game mechanics (trading between their own alts, farming events twice, etc). This mostly lives on the API/DB side, not frontend, but it affects the registration flow's shape so flagging it now:

- new accounts get a 7-day trade lock (can't trade cards/currency with other players until account age >= 7 days) — this is the main mechanism, simple and effective on its own
- API will also capture device fingerprint + IP at registration time for correlation
